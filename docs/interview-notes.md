# 面试准备笔记

## 项目背景

这是一个个人研究项目，用于辅助分析国内公募基金。项目动机：市面上基金分析工具多为券商定制或数据滞后，缺少将技术指标、风险评分、回测验证和 LLM 分析整合到单一工作流的开源工具。

**重要定位：** 这是一个数据采集 + 量化分析 + AI 辅助判断的研究工具，不是投资建议系统，更不是交易信号生成器。

## 技术栈

| 层 | 技术 | 选型理由 |
|---|------|---------|
| 后端 | Python 3.10+ / FastAPI / uvicorn | 异步支持好，类型提示完善，适合数据密集型 API |
| 数据 | AKShare / efinance / Tencent | 国内基金数据开源接口，多源 fallback 提高可用性 |
| LLM | OpenAI 兼容 API | 解耦具体服务商，支持任意兼容服务 |
| 前端 | React 18 + Vite 5 + TypeScript | 类型安全，构建快，生态成熟 |
| 存储 | JSON 文件 + 原子写入 | 轻量、可审计、无需数据库 |
| 测试 | unittest + TestClient | Python 标准库，零额外依赖 |
| CI | GitHub Actions | 免费，与 GitHub 深度集成 |

## 我负责的内容

- 全栈独立开发：从数据采集、指标计算、风险模型到前后端交互
- 多源数据 fallback 策略设计：AKShare → efinance → Tencent 三级降级
- 风险评分引擎：多维度加权模型（趋势 + 回撤 + 波动率 + 持仓 + 宏观）
- 回测验证模块：方向准确率、Brier 分数、概率校准、数据泄露检测
- 前端工程化拆分：从 960 行单文件拆分为 6 个独立组件
- 后端轻量工程化：Pydantic schemas 分离、storage 层抽象、原子写入
- CI/CD 配置 + 测试体系建设

## 技术难点

### 1. 多源数据 Fallback 与容错
- **问题**：国内基金数据接口不稳定，各平台数据格式不一致
- **方案**：三级 fallback（AKShare → efinance → Tencent），每层独立 try/except，数据格式统一清洗，返回 `data_quality` 标识

### 2. 回测数据泄露防护
- **问题**：机器学习中常见的"用未来数据预测过去"的泄露问题
- **方案**：严格按时间切分训练/验证集，在 `backtest_engine.py` 中验证两次独立切分的一致性

### 3. 风险评分可解释性
- **问题**：评分太黑盒，用户无法理解分数含义
- **方案**：多维度分解 + 每维度独立评分 + 权重可追溯，在分析结果中展示各维度贡献

### 4. LLM Fallback 策略
- **问题**：LLM API 不可用或未配置时系统仍要正常工作
- **方案**：LLM 分析完全解耦为可选模块，不可用时自动回退到规则化分析，不影响核心量化指标

## 风险控制设计

整个分析流程中每个环节都有不确定性提示：

- **数据质量标识**：每个数据源返回 `data_quality: good/partial/degraded`
- **回测不确定性列表**：`main_uncertainties` 列出模型可能不可靠的具体原因
- **概率质量评级**：`probability_quality: low/medium/high`，低质量时概率不可信
- **基线对比**：回测结果包含 `rule_vs_best_baseline_edge`，负值表示模型未跑赢简单基线
- **LLM 置信度**：分析结论包含 `confidence: 低/中/高`，低置信度时提示仅供参考
- **免责声明**：前端每个分析结果页都显示"不构成投资建议"

## 面试问答准备

### Q: "这个预测准吗？"

**回答思路：**

> 这个项目不是预测系统。它的回测模块反而是用来验证模型局限性的——当前规则模型在 7 天方向准确率约 17-31%，明显低于"始终判断横盘"的简单基线（~72%）。这说明短期市场方向预测非常困难。
>
> 系统的价值不在于"预测涨跌"，而在于：① 汇总多维数据让用户自己做判断；② 风险评分为持仓者提供量化参考；③ 回测坦诚展示模型局限，避免过度自信。
>
> 我在代码中刻意保留了回测失败的信息展示，而不是只展示好看的数据——因为投资项目中最危险的就是虚假的确定感。

### Q: "你项目最大的工程问题是什么？"

**回答思路：**

> 最大的挑战是如何在"提供有用信息"和"避免误导用户"之间取得平衡。
>
> 技术上我可以在前端显示更激进的买卖建议，回测模块也可以只上报好看的数据。但实际项目中我做了相反的选择：回测模块暴露所有不确定性，LLM 分析标注置信度，风险评分分解为可解释的维度。
>
> 工程上更具体的问题是多源数据一致性——不同数据源对同一基金的净值数据可能有细微差异（如复权方式不同），我做了一层统一清洗，但完全消除差异需要更完善的数据校验层。

### Q: "下一步怎么优化？"

**回答思路：**

> 短期内最有价值的改进：
>
> 1. **数据库持久化**：当前 JSON 文件存储适合原型，但并发写入和查询能力有限。迁移到 SQLite（保持零运维）可解决原子性问题和查询效率。
> 2. **前端可视化**：净值曲线图、风险雷达图、回测校准曲线——当前只有表格和数字，可视化能大幅提升信息密度。
> 3. **回测基线扩展**：当前只对比了"始终横盘"基线，可以增加简单的均线策略、动量策略作为对比基线。
> 4. **Docker 化**：降低环境配置成本，让面试官 clone 后一行命令即可运行。
>
> 长期可以考虑因子模型（Fama-French）替代当前纯规则评分，但收益递减明显——对个人投资者，可解释性比模型复杂度更重要。

### 详细版（适合面试讲解，按 STAR 结构展开）

**S — 项目背景**

个人研究项目，解决市面上基金分析工具的不足：券商定制工具数据滞后，免费工具缺少将技术指标、风险评分、回测验证和 AI 分析整合的工作流。目标是打造一个数据采集 → 量化分析 → AI 辅助判断的完整链路，同时通过回测坦诚展示模型局限，避免虚假确定感。

**T — 我的角色**

独立全栈开发，负责从数据采集、指标计算、风险模型、回测验证到前后端交互的全部设计和实现。

**A — 核心工作**

1. **数据层**：设计三级 fallback 策略（AKShare → efinance → Tencent），每层独立异常处理，数据格式统一清洗，返回 `data_quality` 标识。所有源不可用时返回明确错误而非无效数据。

2. **量化层**：构建多维技术指标体系（涨跌幅、最大回撤、波动率、夏普比率、Calmar 比率、均线趋势），风险评分引擎（加权多维度分解），走势情景分析（1/3/7/30 天概率化判断）。

3. **验证层**：回测模块计算方向准确率、Brier 分数、概率校准曲线，严格按时间切分防数据泄露，对比简单基线（"始终横盘"），当模型未稳定优于基线时主动提示不确定性。

4. **AI 层**：LLM 分析模块完全解耦为可选组件，不可用时自动回退规则化分析，不阻塞核心量化功能。

5. **工程化**：后端 Pydantic schemas 分离 + storage 层抽象 + 原子写入；前端从 960 行单文件拆分为 6 个独立组件；GitHub Actions CI；32 个单元测试；完善的 docs/ 文档目录。

**R — 成果与反思**

- 系统可在无 LLM、无网络条件下正常运行和测试
- 回测数据坦诚展示：当前规则模型 7 天方向准确率约 17-31%，低于"始终横盘"基线（~72%），系统会主动提示这一局限性
- 最大收获：在"提供有用信息"和"避免误导用户"之间找到平衡——每个环节都设计了不确定性提示

**面试中可能被追问的点：**

- **"回测准确率这么低，这个系统有什么价值？"** → 价值不在预测涨跌，而在汇总多维数据 + 风险量化 + 坦诚展示局限。投资中最危险的是虚假确定感。
- **"为什么用 JSON 文件而不是数据库？"** → 原型阶段优先轻量和可审计。迁移到 SQLite 成本低，已预留 storage 抽象层。
- **"LLM 在你的系统里扮演什么角色？"** → 辅助总结，不是决策。LLM 不可用时系统完全正常运行，量化指标不受影响。
- **"你从这个项目中学到什么？"** → 全栈闭环能力、工程化意识、以及最重要的——在技术能力和用户保护之间做取舍的判断力。
